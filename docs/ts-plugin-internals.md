# TS Plugin Internals

> [!NOTE]
> This document was generated by Claude Code and reviewed by a maintainer.

This document describes the internal architecture of CSS Modules Kit's TypeScript Language Service Plugin (`packages/ts-plugin`). It covers how Volar.js is used to create virtual TypeScript files from CSS Modules, how position mappings work, and how TypeScript Language Service results are translated back to CSS positions.

## Overview

The TS plugin makes CSS Modules behave as if they were TypeScript files. It achieves this through:

1. **Volar.js Language Plugin**: Creates virtual `.d.ts` code from `.module.css` files
2. **Position Mappings**: Maintains correspondence between CSS and generated TypeScript positions
3. **Language Service Proxy**: Intercepts TypeScript Language Service responses and enriches them with CSS-specific information

```
┌─────────────────┐     ┌──────────────────────┐     ┌─────────────────────┐
│  a.module.css   │     │  Virtual .d.ts       │     │  TypeScript         │
│                 │────>│  (generated by       │────>│  Language Service   │
│  .a_1 { ... }   │     │   Volar.js)          │     │                     │
│                 │<────│                      │<────│  definitions,       │
│  CSS positions  │     │  TS positions        │     │  references, etc.   │
└─────────────────┘     └──────────────────────┘     └─────────────────────┘
        ^                         |
        |    CodeMapping          |
        └─────────────────────────┘
```

## Virtual Code Generation

### How CSS becomes TypeScript

When a `.module.css` file is opened, the Volar.js Language Plugin (`packages/ts-plugin/src/language-plugin.ts`) creates a `VirtualCode`:

1. The CSS file content is parsed by `parseCSSModule()` (from `@css-modules-kit/core`)
2. The parsed result is passed to `generateDts()` (from `packages/core/src/dts-generator.ts`)
3. The generated `.d.ts` text and position mappings are returned as a `VirtualCode`

### Generated code format

There are two modes: **default export** and **named exports**. The mode depends on the `namedExports` configuration option.

#### Default export mode (`namedExports: false`)

```css
/* a.module.css */
.a_1 {
  color: red;
}
.a_2 {
  color: blue;
}
```

Generates:

```ts
// @ts-nocheck
declare const styles = {
  'a_1': '' as readonly string,
  'a_2': '' as readonly string,
};
export default styles;
```

Note that property names are **single-quoted**. This is important for the mapping behavior described later.

#### Named exports mode (`namedExports: true`)

```css
/* a.module.css */
.a_1 {
  color: red;
}
.a_2 {
  color: blue;
}
```

Generates:

```ts
// @ts-nocheck
export var a_1: string;
export var a_2: string;
```

In named exports mode, property names are **not quoted**.

#### Import handling

`@import` in default export mode:

```css
@import './b.module.css';
```

Generates:

```ts
// @ts-nocheck
function blockErrorType<T>(val: T): [0] extends [(1 & T)] ? {} : T;
declare const styles = {
  ...blockErrorType((await import('./b.module.css')).default),
};
```

`@value` imports in default export mode:

```css
@value b_1, b_2 from './b.module.css';
@value c_1 as aliased_c_1 from './c.module.css';
```

Generates:

```ts
declare const styles = {
  'b_1': (await import('./b.module.css')).default['b_1'],
  'b_2': (await import('./b.module.css')).default['b_2'],
  'aliased_c_1': (await import('./c.module.css')).default['c_1'],
};
```

## Position Mappings (CodeMapping)

### CodeMapping structure

A `CodeMapping` connects positions in the source CSS file to positions in the generated `.d.ts` file:

```ts
interface CodeMapping {
  sourceOffsets: number[]; // Positions in the .module.css file
  lengths: number[]; // Lengths of tokens in the .module.css file
  generatedOffsets: number[]; // Positions in the generated .d.ts file
  generatedLengths?: number[]; // Lengths in the .d.ts file (defaults to `lengths` if omitted)
}
```

Each index `i` defines a mapping pair: `sourceOffsets[i]` in CSS corresponds to `generatedOffsets[i]` in TypeScript, with lengths `lengths[i]` and `generatedLengths[i]` respectively.

### Mapping examples

#### Named exports mode (simple case)

```
a.module.css:
1 | .a_1 { color: red; }
    ^ sourceOffsets[0] = 1, lengths[0] = 3

a.module.css.d.ts:
1 | // @ts-nocheck
2 | export var a_1: string;
               ^ generatedOffsets[0] = 26
```

Mapping: `{ sourceOffsets: [1], generatedOffsets: [26], lengths: [3] }`

Since `generatedLengths` is omitted, it defaults to `lengths` (`[3]`). Both the source and generated tokens have the same length.

#### Default export mode (with single quotes)

```
a.module.css:
1 | .a_1 { color: red; }
    ^ sourceOffsets[0] = 1, lengths[0] = 3

a.module.css.d.ts:
1 | // @ts-nocheck
2 | declare const styles = {
3 |   'a_1': '' as readonly string,
       ^ generatedOffsets[0] = 27 (after the opening quote)
```

The mapping points to the token name inside the quotes; the surrounding quotes are not covered:

```
mapping: { sourceOffsets: [1], generatedOffsets: [27], lengths: [3] }
```

### Mapping registration with Volar.js

The mapping is registered with Volar.js through the `VirtualCode.mappings` array:

```ts
// packages/ts-plugin/src/language-plugin.ts
return {
  id: 'main',
  languageId: 'typescript',
  snapshot: {
    /* ... */
  },
  mappings: [mapping].map((mapping) => ({ ...mapping, data: { navigation: true } })),
  linkedCodeMappings: [{ ...linkedCodeMapping, data: undefined }],
};
```

The `data: { navigation: true }` flag enables navigation features (Go to Definition, Find References, Rename) for this mapping.

## How Volar.js Translates Positions

### translateOffset

The core of Volar.js's position mapping is the `translateOffset` function (`@volar/source-map/lib/translateOffset.js`).

Given a position in one coordinate space, it finds the corresponding position in another:

```
translateOffset(start, fromOffsets, toOffsets, fromLengths, toLengths, preferEnd)
```

**Algorithm:**

1. Binary search to find the mapping segment containing `start`
2. Check: `start >= fromOffsets[i] && start <= fromOffsets[i] + fromLengths[i]`
3. Calculate relative position: `relativePos = start - fromOffsets[i]`
4. Scale to target: `rangeOffset = Math.min(relativePos, toLengths[i])`
5. Return: `toOffsets[i] + rangeOffset`

**The `preferEnd` parameter:** When `preferEnd` is true, `toLength > fromLength`, and `relativePos === fromLength` (at the exact end of source range), `rangeOffset = toLength` is used instead. This ensures the end of an expanded range maps correctly.

**Example (simple):**

```
translateOffset(28, [26], [1], [3], [3])
  → relativePos = 28 - 26 = 2
  → rangeOffset = Math.min(2, 3) = 2
  → result = 1 + 2 = 3
```

**Example (with different lengths):**

```
translateOffset(27, [26], [1], [5], [3])
  → relativePos = 27 - 26 = 1
  → rangeOffset = Math.min(1, 3) = 1
  → result = 1 + 1 = 2
```

This is the problematic case described in the [Single-Quote Span Problem](#the-single-quote-span-problem) section.

### SourceMap.findMatchingOffsets

This generator function finds all mappings that match a given offset:

1. Uses a pre-computed memo index (built by `createMemo`) for O(log n) lookup
2. Binary searches memo offsets to find candidate mappings
3. For each candidate mapping, calls `translateOffset`
4. Yields `[mappedOffset, mapping]` for each successful translation

### SourceMap.findMatchingStartEnd

This function translates a range (start + end) from one coordinate space to another:

**Phase 1 (ideal):** Find a single mapping that contains both `start` and `end`.

- For each mapping that matches `start`, try to translate `end` within the same mapping
- Yields `[mappedStart, mappedEnd, mapping, mapping]`

**Phase 2 (fallback):** If no single mapping covers both start and end, and `fallbackToAnyMatch` is true:

- Use different mappings for start and end
- Yields `[mappedStart, mappedEnd, mappingStart, mappingEnd]`
- Only if `mappedEnd >= mappedStart`

## The Single-Quote Span Problem

### Background

TypeScript's Language Service returns **different spans** for quoted property names across different APIs:

| API                       | Span for `'a_1'`           | Includes quotes? |
| ------------------------- | -------------------------- | ---------------- |
| `getDefinitionAtPosition` | `{ start: 26, length: 5 }` | Yes              |
| `findReferences`          | `{ start: 27, length: 3 }` | No               |
| `findRenameLocations`     | `{ start: 27, length: 3 }` | No               |

This is an inherent behavior of TypeScript itself. When a property name is quoted (like `'a_1'`), `getDefinitionAtPosition` returns a span that includes the surrounding quotes, while `findReferences` and `findRenameLocations` return spans that only cover the identifier inside the quotes.

### The problem

Consider this generated code with a single mapping:

```ts
declare const styles = {
  'a_1': string,
};
```

```
Mapping: { generatedOffsets: [27], lengths: [3], sourceOffsets: [1] }
                              ^ points to 'a' (after the opening quote)
```

- **`findReferences`** returns `{ start: 27, length: 3 }` for `a_1`. `translateOffset(27, [27], [1], [3], [3])` → returns `1`. Correct!

- **`getDefinitionAtPosition`** returns `{ start: 26, length: 5 }` for `'a_1'`. `translateOffset(26, [27], ...)` → `26 < 27`, so no mapping found. Returns `undefined`. Volar.js falls back to `{ start: 0, length: 0 }`. Broken!

### Why a single mapping including quotes also fails

If we map `'a_1'` instead (including quotes):

```
Mapping: { generatedOffsets: [26], lengths: [3], sourceOffsets: [1], generatedLengths: [5] }
                              ^ points to opening quote
```

- **`getDefinitionAtPosition`** returns `{ start: 26, length: 5 }`. `translateOffset(26, [26], [1], [5], [3])` → `relativePos = 0` → returns `1`. Correct!

- **`findReferences`** returns `{ start: 27, length: 3 }`. `translateOffset(27, [26], [1], [5], [3])` → `relativePos = 1` → `rangeOffset = Math.min(1, 3) = 1` → returns `1 + 1 = 2`. Wrong! Expected `1`.

The issue is that `findReferences` returns the position of `a` (offset 27), which is 1 character past the mapping's start. The `translateOffset` function proportionally maps this to `sourceOffset + 1 = 2`, but the correct source position is `1`.

### Why overlapping mappings in a single object don't work

Putting both ranges in the same CodeMapping object:

```
Mapping: {
  generatedOffsets: [26, 27],
  lengths: [3, 3],
  sourceOffsets: [1, 1],
  generatedLengths: [5, 3]
}
```

This doesn't work because Volar.js does not support overlapping ranges within a single mapping. The ranges `[26, 26+5)` and `[27, 27+3)` overlap, and Volar.js's binary search finds the first match, which may be the wrong one.

Reference: https://github.com/volarjs/volar.js/issues/203

### The solution: a custom mapper that strips outer quotes on fallback

As a rule, CSS Modules Kit registers mappings that **do not include the surrounding quotes**, and instead swaps Volar.js's default mapper for a custom one that retries with outer characters stripped when the direct lookup fails. The custom mapper is installed by overriding `language.mapperFactory`:

```ts
// packages/ts-plugin/src/index.cts
language.mapperFactory = (mappings) => new CustomSourceMap(mappings);
```

`CustomSourceMap` extends `@volar/language-core`'s `SourceMap` and overrides only `toSourceRange`:

```ts
// packages/ts-plugin/src/source-map.ts
export class CustomSourceMap extends SourceMap<CodeInformation> {
  override *toSourceRange(start, end, fallbackToAnyMatch, filter) {
    let matched = false;
    for (const result of super.toSourceRange(start, end, fallbackToAnyMatch, filter)) {
      matched = true;
      yield result;
    }
    if (matched) return;

    // The outer characters may be surrounding quotes (e.g. `'a_1'`).
    // Retry with them stripped so the inner token name's mapping can match.
    if (end - start >= 2) {
      yield* super.toSourceRange(start + 1, end - 1, fallbackToAnyMatch, filter);
    }
  }
}
```

For each API:

- **`findReferences`** / **`findRenameLocations`** (start=27, length=3): the inner mapping matches directly → `translateOffset(27, [27], [1], [3], [3])` → returns `1`. Correct!
- **`getDefinitionAtPosition`** (start=26, length=5): the inner mapping doesn't match (26 < 27). The fallback retries with `toSourceRange(27, 29, …)`, which matches → returns `1`. Correct!

Reference: https://github.com/mizdra/volar-single-quote-span-problem

## LinkedCodeMappings

### Purpose

`linkedCodeMappings` enable bidirectional navigation between related positions **within the same generated file**. They are used for features like "Find All References" on aliased imports.

### Example

```css
/* a.module.css */
@value c_1 as aliased_c_1 from './c.module.css';
```

Generates (default export mode):

```ts
declare const styles = {
  'aliased_c_1': (await import('./c.module.css')).default['c_1'],
};
```

The `linkedCodeMapping` connects `'aliased_c_1'` and `'c_1'` in the generated code:

```
linkedCodeMapping: {
  sourceOffsets: [<offset of 'aliased_c_1'>],     // In generated .d.ts
  lengths: [<length of 'aliased_c_1' + quotes>],
  generatedOffsets: [<offset of 'c_1'>],           // In generated .d.ts
  generatedLengths: [<length of 'c_1' + quotes>],
}
```

Note: Despite the names `sourceOffsets` and `generatedOffsets`, both refer to positions within the same generated `.d.ts` file. The names are interchangeable for `linkedCodeMappings`.

**Why the ranges include the surrounding quotes:** When `getDefinitionAtPosition` is invoked on a quoted property name, TypeScript returns a `textSpan` that includes the surrounding single quotes (see [The Single-Quote Span Problem](#the-single-quote-span-problem)). To resolve links between related positions, Volar.js feeds that raw `textSpan.start` into `LinkedCodeMap.getLinkedOffsets` — and this path does **not** go through `mapperFactory`, so the `CustomSourceMap` quote-strip fallback cannot help here. For the linked-code lookup to match, the offsets stored in `linkedCodeMappings` must themselves cover the opening quote.

### How LinkedCodeMap works

The `LinkedCodeMap` class (in `@volar/language-core`) extends `SourceMap` and provides `getLinkedOffsets(start)`:

```ts
class LinkedCodeMap extends SourceMap {
  *getLinkedOffsets(start) {
    for (const mapped of this.toGeneratedLocation(start)) {
      yield mapped[0];
    }
    for (const mapped of this.toSourceLocation(start)) {
      yield mapped[0];
    }
  }
}
```

It searches both directions (source→generated and generated→source) to find all linked positions. This enables features like:

- "Find All References" on `aliased_c_1` also finds `c_1`
- "Find All References" on `c_1` also finds `aliased_c_1`

## Language Service Proxy

The TS plugin proxies several TypeScript Language Service methods to enhance CSS Modules support (`packages/ts-plugin/src/language-service/proxy.ts`).

### How Volar.js proxies the Language Service

Before the CSS Modules Kit proxy, Volar.js already wraps the TypeScript Language Service. The Volar.js wrapper (`@volar/typescript/lib/node/transform.ts`) automatically:

1. **Translates input positions**: Converts source file positions to generated code positions using `toGeneratedOffset`
2. **Calls the underlying TypeScript Language Service** with generated code positions
3. **Translates output spans**: Converts generated code spans back to source file spans using `toSourceRanges` / `transformTextSpan`

This means the CSS Modules Kit proxy receives results that are **already translated** to CSS file positions by Volar.js.

### What the CSS Modules Kit proxy adds

The proxy in `packages/ts-plugin/src/language-service/proxy.ts` adds CSS-specific enhancements on top of Volar.js's translation:

| Method                                           | Enhancement                                                                            |
| ------------------------------------------------ | -------------------------------------------------------------------------------------- |
| `getDefinitionAndBoundSpan`                      | Adds `contextSpan` for Definition Preview (shows the full CSS rule)                    |
| `findReferences`                                 | Merges `ReferencedSymbol`s that share the same definition                              |
| `getCompletionsAtPosition`                       | Prioritizes `styles` import, converts `className` to JSX format, filters named exports |
| `getCompletionEntryDetails`                      | Converts default imports to namespace imports for CSS Modules                          |
| `getSyntacticDiagnostics`                        | Reports parse-time diagnostics that do not overlap with the standard CSS LS            |
| `getSemanticDiagnostics`                         | Reports CSS Modules validation errors (unknown tokens, unresolvable imports)           |
| `getCodeFixesAtPosition`                         | Adds "Fix Missing CSS Rule" quick fix                                                  |
| `getApplicableRefactors` / `getEditsForRefactor` | Adds "Create CSS Module File" refactor                                                 |

In particular, `getSyntacticDiagnostics` does **not** report generic invalid CSS syntax. `parseCSSModule()` is called with `includeSyntaxError: false`, because the standard CSS Language Server already reports those errors. The diagnostics returned here are parse-time errors specific to CSS Modules syntax, such as malformed `:local` or `@value`.

### Protocol handlers

Some features are implemented as custom protocol handlers rather than relying only on the standard editor-to-LanguageService path. In VS Code, the extension forwards requests directly to tsserver using custom commands ("Request Forwarding to tsserver"), mainly to avoid conflicts with the standard CSS Language Server:

| Handler                         | Purpose                                                                        |
| ------------------------------- | ------------------------------------------------------------------------------ |
| `_css-modules-kit:rename`       | Calls `findRenameLocations` through tsserver for CSS rename                     |
| `_css-modules-kit:renameInfo`   | Calls `getRenameInfo` directly                                                 |
| `_css-modules-kit:documentLink` | Returns import specifier positions as document links                           |

These handlers do **not** bypass CSS Modules Kit's own proxied Language Service. They call `project.getLanguageService()`, and that service has already been wrapped by both Volar.js and CSS Modules Kit during plugin setup.
